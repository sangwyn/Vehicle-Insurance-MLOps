# Постановка задачи и описание решения

## 1. Постановка задачи

Цель проекта: построить MLOps-пайплайн для работы с потоковыми данными по автострахованию и автоматизировать запуск пайплайна через CI/CD.

Требования:
- обрабатывать данные батчами;
- обучать/дообучать модель прогнозирования страховой выплаты;
- сохранять логи и артефакты;
- поддерживать сценарии `inference`, `update`, `summary`.

## 2. Данные

Источники данных:
- `data/motor_data11-14lats.csv`
- `data/motor_data14-2018.csv`

Тип данных: табличные записи по страховым полисам и объектам страхования.

Целевая переменная:
- `CLAIM_PAID` — фактическая страховая выплата (регрессия).

## 3. Полный pipeline

### Этап 1. Data Collection

- чтение источников и эмуляция потока по батчам;
- запись сырых данных в SQLite (`raw_data.db`);
- сохранение batch-метаданных (`batch_meta`), логирование.

### Этап 2. Data Analysis

- оценка data quality:
  - completeness;
  - validity;
  - timeliness;
- очистка данных:
  - удаление дубликатов;
  - обработка пропусков;
  - базовые проверки допустимости значений;
- EDA и сохранение отчётов в `logs/anal_report_*`.

### Этап 3. Data Preparation

- feature engineering:
  - `insr_duration`,
  - `insr_begin_season`,
  - `insr_end_season`,
  - `prem_insr_ratio`;
- формирование признакового пространства:
  - числовые признаки: импутация + масштабирование (`standard`/`minmax`);
  - категориальные признаки: импутация + one-hot encoding;
- сохранение:
  - подготовленных данных (`storage/prepared.parquet` или fallback `storage/prepared.csv`);
  - препроцессора (`storage/preprocessor.joblib`).

### Этап 4. Model Training / Finetuning

Рассматриваемые модели:
- `lr` (SGDRegressor, как линейная модель в online-режиме),
- `knn`,
- `dt` (DecisionTreeRegressor).

Гиперпараметры:
- базовые задаются в `configs/default.yaml`;
- при включенном `search_mode` используется GridSearchCV.

Дообучение:
- для моделей с `partial_fit` используется инкрементальный шаг;
- для `DecisionTreeRegressor` реализовано дообучение без полного переобучения:
  - фиксируется структура дерева;
  - обновляются статистики листьев на новом батче.

### Этап 5. Model Validation

- схемы валидации:
  - hold-out;
  - CV (KFold);
- метрики:
  - MAE,
  - MSE / RMSE,
  - R2;
- quality gate по порогам из конфигурации;
- реестр версий модели в `logs/model_registry/tree`.

### Этап 6. Артефакты

Сохраняются:
- логи (`logs/`);
- обученные/дообученные модели (`model/`, `results/`);
- отчёты по quality, обучению, обновлению и summary.

### Этап 7. Управление системой (`main.py`)

Поддерживаемые режимы:
- `inference` — получить предсказания и сохранить CSV с `predict`;
- `update` — выполнить дообучение модели и сохранить отчёт `update_report_*`;
- `summary` — сформировать агрегированный мониторинговый отчёт по истории запусков.

## 4. CI/CD workflow

Выбранный подход:
- GitHub Actions workflow `.github/workflows/train.yml`;
- триггеры: `push`, `pull_request`, ручной запуск (`workflow_dispatch`);
- шаги:
  - checkout;
  - установка Python и зависимостей;
  - запуск `full_pipeline.py`;
  - выгрузка артефактов (логи и модель).

Аргументация:
- покрывает обязательные критерии задания;
- легко проверяется и воспроизводится;
- позволяет расширить workflow до CRON-инкрементального сценария.
